# Visión del proyecto

## ¿Qué es UptimeGuard?

UptimeGuard es un proyecto **didáctico de DevOps** cuyo objetivo es enseñar cómo diseñar, desplegar y operar un sistema real de monitorización de servicios.

El proyecto se centra en **aprender haciendo**, construyendo un sistema funcional que realiza checks HTTP periódicos sobre URLs y envía alertas cuando detecta caídas o recuperaciones.

No es un producto comercial ni un SaaS, sino una **base de aprendizaje práctica** para personas que están formándose en DevOps, Cloud o SRE.

---

## Problema que aborda

Muchas personas que aprenden DevOps:
- conocen herramientas aisladas (Docker, CI, Linux),
- pero no saben cómo se integran en un **sistema completo**,
- ni cómo se opera un servicio real en producción.

Además, la mayoría de ejemplos educativos:
- son demasiado simples,
- no reflejan patrones reales,
- o no incluyen operación, alertas ni observabilidad.

UptimeGuard existe para cubrir ese vacío.

---

## Público objetivo

Este proyecto está pensado principalmente para:

- Personas que están aprendiendo **DevOps / Cloud / SRE**
- Desarrolladores junior que quieren entender la parte operativa
- Estudiantes o bootcamps que necesitan un proyecto final realista
- Personas que quieren practicar CI/CD, Docker y operación real

---

## Propuesta de valor

UptimeGuard permite aprender DevOps construyendo un sistema que:

- se despliega con Docker
- tiene separación clara de responsabilidades
- incluye automatización desde el inicio
- genera alertas reales ante fallos
- se puede ejecutar en local o en un VPS

Todo el proyecto está diseñado para ser:
- simple
- entendible
- extensible
- documentado

---

## Qué es y qué no es este proyecto

### Es:
- Un proyecto educativo
- Un sistema funcional y usable
- Una base para experimentar y extender
- Un ejemplo realista de arquitectura DevOps

### No es:
- Un SaaS comercial
- Un reemplazo de herramientas profesionales de monitorización
- Un proyecto con UI compleja o multiusuario
- Un sistema pensado para escalar masivamente

---

## Filosofía del proyecto

- Primero entender, luego optimizar
- Primero algo que funcione, luego mejorar
- Documentación como parte del producto
- Decisiones técnicas explicadas, no ocultas

El foco no está en la herramienta, sino en el **razonamiento detrás de cada decisión**.
